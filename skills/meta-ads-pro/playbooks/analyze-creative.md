# Аудит статичного креативу (image ad)

> **Призначення.** Дати по ОДНОМУ статичному оголошенню (image ad) дієвий вердикт: **Risk Score (0–100)** + чек-ліст по 6 критеріях + конкретні фікси (паузити / переробити / лишити + які creative enhancements перемкнути). Не «покращити креатив», а «overlay `add_overlays` ON, бо тексту в зображенні немає; перенести субтитр з нижнього краю в центр під кроп 4:5; паузити ad_id=X (CPL 3× target за 7д, n=4 конв.)».
>
> **Два режими.** (А) **Pre-flight** — креатив ще не запущений, даних немає → аудит лише по макету/правилах (Risk Score без performance-блоку). (Б) **Live** — оголошення крутиться → той самий чек-ліст + performance-діагностика з `get_insights` (era-валідна, з порогами значущості).
>
> **Залізне правило.** KNOWLEDGE (принцип з `knowledge/`) ≠ DATA (цифра з Marketing API). Принцип каже «один angle на оголошення»; чи виконано це в КОНКРЕТНОМУ креативі — встановлюється з `get_ad_creatives` (body/title/image), а не з голови. Кожна цифра у звіті — з джерела (ad_id / період / метрика).

## 0. Коли застосовувати

- Користувач дав 1 креатив (зображення + текст або ad_id) і просить «оціни / що не так / чому не працює / before-launch перевірка».
- Перед масштабуванням креативу (дублювання з бюджетом) — швидкий risk-gate.
- У межах `audit-account` — як дрилл-даун на «найгірший за CPL» / «single point of failure» ad.

Якщо креативів багато — застосовувати цей playbook **по одному**; зведення робити окремо (як у `audit-account` секції «creative concentration»).

## 1. Збір даних (read-before-write)

### 1.1 Що завжди читаємо (обидва режими)

```
get_ads(adset_id=...)                         # знайти ad_id, status, name
get_ad_creatives(ad_id=...)                   # body, title (headline), description,
                                              #   image_url/object_story_spec,
                                              #   degrees_of_freedom (які enhancements ON/OFF),
                                              #   call_to_action_type
```

- **Якщо є зображення-файл/URL** — відкрити й подивитись очима (Read зображення). Без перегляду самого макета аудит «читабельності тексту» і «кропу» неможливий → так і сказати: «потрібен файл креативу, інакше п.3–п.6 чек-листа не оцінюються».
- З `get_ad_creatives` витягнути блок **enhancements** (у Marketing API це `degrees_of_freedom_spec` / `creative_features_spec`): які саме увімкнені — `standard_enhancements`, `image_templates`, `text_optimizations`, `image_touchups`, `add_text_overlay`, `enhance_cta`, `image_uncrop`, `inline_comment` тощо. Це критично для п.5 (creative enhancements).

### 1.2 Що додатково читаємо в режимі (Б) Live

```
get_insights(object_id=ad_id, level="ad", time_range="last_7d")
get_insights(object_id=ad_id, level="ad", time_range="last_30d")
# опційно, якщо треба перевірити гіпотезу «формат/плейсмент б'є»:
get_insights(object_id=ad_id, level="ad", time_range="last_30d",
             breakdowns="publisher_platform")     # FB feed / IG feed / Reels / Stories / AN
get_insights(object_id=ad_id, level="ad", time_range="last_30d",
             breakdowns="age")  # + окремо gender — перевірити ICP-fit за фактом
```

**Поріг значущості (з `safety_rules` + §9 META-PROMPT):** не виносити вердикт по перформансу, якщо **<1000 показів** АБО **<3 конверсій** АБО **<2 днів** відкрутки. Менше — статус `INSUFFICIENT_DATA`: «потрібно ще покази/конверсії, не змінювати». Те саме для breakdown-зрізів: зріз із <300 показів — не основа для висновку «Stories гірші за Feed», а лише гіпотеза.

> ⚠️ **Не плутати креатив із таргетингом.** Поганий CPL ще не означає поганий креатив — це може бути аудиторія/оффер/LP. Креатив-діагностика чесна тоді, коли інші змінні зафіксовані (один adset, broad, той самий оффер). Якщо ні — пиши «CPL міксує креатив+аудиторію, для чистого тесту креативу потрібен роздільний спліт → [[12-testing/post-andromeda-testing]]».

## 2. Шість критеріїв аудиту (ядро)

Кожен критерій: **що перевіряємо → джерело сигналу → як оцінюємо → бали ризику**. Бали підсумовуються в Risk Score (розділ 3). Принципи — з `knowledge/`; факт «виконано/ні» — з конкретного креативу.

### Критерій 1 — Один чіткий ad angle (не «все в одному») · вага до 30

**Принцип [P]:** один angle на одне оголошення; надлишок інформації вбиває рекламу — проспект не встигає зрозуміти «що це, що робить, чи мені підходить». Автор бачить НАБАГАТО більше задто складних оголошень, ніж задто простих. ⟦[[06-creatives/why-ads-fail]]⟧

**Сигнал (з `get_ad_creatives`):** body + headline + текст-на-зображенні. Порахувати, **скільки різних причин купити** заявлено одночасно.

| Стан креативу | Бали |
|---|---:|
| Один наскрізний angle (зображення + headline + body про одне) | 0 |
| Домінує один angle, але є 1 «зайва» теза | +10 |
| 2–3 рівноправні angle злиті в одному оголошенні | +20 |
| «Все в одному»: список фіч/бенефітів, кілька офферів, message overload | +30 |

**Прапор «message overload»** піднімати, якщо оголошення намагається сказати все одразу. Фікс — не «спростити» абстрактно, а: «винести angle B і angle C у ОКРЕМІ оголошення; у цьому лишити лише angle A (той, що в headline)». Який angle лишати першим — не вгадувати: «найсильніший angle визначається тестом, а не інтуїцією» ⟦[[06-creatives/why-ads-fail]]⟧ → план тесту angles у розділі 4.

### Критерій 2 — Перші секунди уваги (stop-the-scroll) · вага до 20

**Принцип [P]:** велика частина задачі креативу — привернути увагу в перші моменти; саме тому Meta дає рух (animation) і яскравість. ⟦[[06-creatives/creative-enhancements]]⟧ Для статики «перші секунди» = **перший погляд** (≈0.5–1 c у стрічці): чи є візуальний/смисловий гачок, що зупиняє скрол.

**Сигнал:** перегляд зображення очима + headline.

Чек (кожне «ні» додає бали):
- Є чіткий фокусний елемент / контраст / обличчя / несподіванка, що ловить око у стрічці? → ні: +8
- Headline читається з першого погляду й передає суть/інтригу (а не загальник)? → ні: +7
- Креатив не зливається з органічним фідом у «банерну сліпоту» (надто стоковий/корпоративний)? → ні: +5

**Підтвердження даними (режим Б, якщо n≥1000 показів):** низький **CTR (link)** — кількісний сигнал слабкого гачка/першого враження. Порівнювати з іншими ad у тому ж adset, не з абсолютом. Низький CTR при норм. CPM → проблема гачка/креативу, а не доставки. *(Кількісний поріг CTR не вигадувати — брати медіану по акаунту/adset.)*

### Критерій 3 — Читабельність тексту-overlay · вага до 15

**Принцип [P]:** text-heavy зображення — особливий випадок; для нього animation треба ВИМИКАТИ (текст має бути стабільним для читання), а add_overlays — теж вимикати (бо текст уже є). ⟦[[06-creatives/creative-enhancements]]⟧ Текст у статиці працює лише якщо його реально можна прочитати за пів секунди на мобільному.

**Сигнал:** перегляд макета.

Чек (кожне «ні» додає бали):
- Текст-overlay великий і контрастний до фону (читається на телефоні без зуму)? → ні: +6
- Тексту небагато (1 теза/оффер, а не абзац)? → ні: +5 *(пов'язано з критерієм 1 — багато тексту часто = message overload)*
- Текст НЕ перекриває обличчя/ключовий об'єкт і не тоне в «шумному» фоні? → ні: +4

> Окрема перевірка для **подальших enhancements**: якщо креатив text-heavy — це автоматично диктує OFF для `add_animation` і `add_overlays` (див. критерій 5). Тобто читабельність і enhancements зчеплені.

### Критерій 4 — Відповідність ICP (avatar-fit) · вага до 20

**Принцип [P]:** спершу клієнт, потім angles — обраний ICP диктує, який angle і меседж узагалі доречні; для різних аватарів — різний angle-переможець (SMB ≠ enterprise). Ніша > широта. ⟦[[06-creatives/why-ads-fail]]⟧

**Сигнал:** креатив (хто на зображенні, тон, мова бенефіту) + бриф (ICP/avatar). **DATA-перехресна перевірка (режим Б):** `get_insights breakdowns=age` / `gender` — чи конвертить саме той сегмент, на який цілений креатив.

| Стан | Бали |
|---|---:|
| Креатив явно «говорить» до одного конкретного аватара з брифу (видно «це для такого, як я») | 0 |
| Аватар вгадується, але меседж розмитий / надто широкий («власники бізнесу», «всі, хто…») | +12 |
| Креатив під один аватар, а angle/бенефіт — з мотивацій ІНШОГО сегмента (mismatch) | +20 |

**Якщо брифу/ICP немає** — це сам по собі корінь майбутнього провалу: «перед оцінкою креативу потрібен артефакт ICP (конкретний аватар + критерії); без нього неможливо сказати, чи angle доречний» ⟦[[06-creatives/why-ads-fail]]⟧. Статус критерію → `NEEDS_ICP`, у звіт як перший data-gap.

### Критерій 5 — Правильність creative enhancements · вага до 15

**Принцип [P]:** рекламодавець — **фільтр**: вмикати лише ті enhancement-и, що пасують ЦЬОМУ креативу, вимикати ті, що спотворюють меседж; набір опцій різний для image і video; обов'язково перевіряти AI-music і AI-image variations руками. ⟦[[06-creatives/creative-enhancements]]⟧

**Сигнал:** блок enhancements із `get_ad_creatives` (які ON/OFF) + характер ассета (text-heavy / продуктове фото / фото конкретної особи).

Звірити по матриці (кожне порушення додає бали; макс +15):

| Enhancement | Дефолт | Коли має бути ON | Коли MUST OFF | Порушення → бали |
|---|---|---|---|---:|
| `add_overlays` (текст-оверлеї) | ON | тексту в зображенні немає (продуктове фото) | текст уже є в зображенні (text-heavy) → стане messy | +4 |
| `add_animation` (рух статиці) | ON | візуальне фото без тексту | text-heavy (текст «їде», важче читати) | +4 |
| `enhanced_media_text` (beta, заміна тексту в медіа) | OFF | простий продукт з 1 benefit-оверлеєм | критичний текст: testimonial/точний меседж | +3 |
| `image_templates` / AI image variations | — | варіації кольору/фону «виглядають добре» | офер центрований на КОНКРЕТНІЙ особі, а варіація міняє людину/спотворює фото | +4 |
| `music` | ON | додатковий гачок для тих, хто зі звуком | дефолтний «super corporate» трек не пасує vibe | +2 *(перевір руками; малий важіль)* |
| `enhanced_cta` (цикл CTA) | OFF→рек. ON для image | нестандартний CTA може виділитись | — (низький ризик; за бажанням customize+перегляд) | 0 |
| `visual_touch_ups` / `flex_media` / `text_improvements` | ON | майже завжди лишати ON | рідкісні крайні випадки | 0 (лишати ON) |

**Окрема перевірка — «зруйнований меседж»:** якщо на креативі з критичним текстом (testimonial / точна цифра / конкретний меседж) увімкнено `enhanced_media_text` або `add_overlays` поверх наявного тексту → це руйнує суть, **MUST OFF**, +додатково підсвітити червоним у звіті. ⟦[[06-creatives/creative-enhancements]]⟧

### Критерій 6 — Формат під кроп (safe-zone) · вага до 10

**Принцип [P]:** Meta кропить/розтягує креатив під 4:5 та інші формати й плейсменти (visual touch-ups / flex media / uncrop), тому **проєктуй креатив під це**: важлива інфо/текст/субтитри — по ЦЕНТРУ, не скраю (не вгорі/внизу), інакше при кропі зникне. Розподіл показів: square (feeds) і vertical (reels/stories) — багато, horizontal — мало. ⟦[[06-creatives/creative-enhancements]]⟧

**Сигнал:** перегляд макета + які aspect-ratio файли завантажені (з `get_ad_creatives` / asset_feed).

Чек (кожне «ні» додає бали):
- Ключова інфо/текст/CTA лежить у центральній safe-zone (не у верхніх/нижніх ~15% і не впритул до країв)? → ні: +5
- Є окремі файли під **square (1:1 / 4:5)** І **vertical (9:16)** — або хоча б свідомо покладаємось на uncrop/flex media? → ні: +3
- Креатив не «ламається» при кропі в 4:5 (нічого важливого не відрізає)? → ні: +2

**Підтвердження даними (режим Б):** `get_insights breakdowns=publisher_platform` — якщо на vertical-плейсментах (Reels/Stories) CTR/CR помітно гірші при достатньому n (≥1000 показів на зрізі) → гіпотеза «креатив не адаптований під vertical-кроп». Якщо n малий — лише гіпотеза, не вердикт.

## 3. Risk Score (0–100) і вердикт

```
Risk Score = sum(бали критеріїв 1..6)
           = angle(≤30) + attention(≤20) + readability(≤15)
           + icp_fit(≤20) + enhancements(≤15) + crop(≤10)
```
> Максимум сирих ваг = 110; за наявності рідкісних накладок капувати на **100**.

### Класи ризику та дія

| Risk Score | Клас | Іконка | Вердикт і дія |
|---|---|---|---|
| 0–25 | LOW | 🟢 | Готовий до запуску / до масштабування. Дрібні фікси (зазвичай enhancements-toggle). |
| 26–50 | MEDIUM | 🟡 | Запускати/лишати можна, але з пріоритетними фіксами; для масштабу — спершу полагодити. |
| 51–75 | HIGH | 🟠 | Не масштабувати. Переробити топ-2 критерії перед (пере)запуском. У live: кандидат на заміну. |
| 76–100 | CRITICAL | 🔴 | Не запускати / у live — паузити (за умови порогу значущості). Креатив переробляти з нуля по чек-листу. |

### Накладання з performance-вердиктом (тільки режим Б, n достатній)

Risk Score (якість макета) і performance (CPL/CTR) — **дві різні осі**. Зводимо:

| | Risk Score LOW–MED | Risk Score HIGH–CRIT |
|---|---|---|
| **CPL ≤ target** | 🟢 Лишити; кандидат на дублювання з бюджетом *(крок ≤+30%, див. [[04-budget-scaling/small-budget]] / [[04-budget-scaling/big-spend-scaling]])* | 🟡 Працює попри слабкий макет — лишити, але підготувати чистішу версію (ризик швидкого вигоряння) |
| **CPL > 3× target** (n≥3 конв.) | 🟠 Макет ок — причина деінде: перевірити аудиторію/оффер/LP, не вбивати креатив наосліп | 🔴 **Паузити ad_id=X** (CPL Nх target за 7д, n=K конв.) + переробити по топ-2 критеріях |

> **Anti-hallucination guard.** Якщо n<1000 показів або <3 конв. — performance-клітинку НЕ заповнювати: «INSUFFICIENT_DATA, рішення про паузу передчасне, чекати». Вердикт «паузити» з performance-причини **вимагає** ad_id + період + CPL + кількість конверсій. Без цього паузу не рекомендувати (правило ad-eater з `safety_rules`: CPL>3× target → пауза, але після порогу значущості).

## 4. Фікси (дієві, з прив'язкою до об'єкта)

Кожен фікс — конкретна дія над конкретним об'єктом. Формат: **{критерій} → {object_id} → {дія} → {очікуваний ефект} → {як перевірити}**. Категорії:

**A. Enhancements-toggle (миттєво, без перероблення макета).** Найдешевший клас фіксів — змінюється на рівні ad creative, не потребує нового файлу.
- *Приклад:* «Критерій 5 → ad_id=X → `add_overlays` OFF (на креативі вже є текст у зображенні, overlay робить messy) ⟦[[06-creatives/creative-enhancements]]⟧ → очікувано: чистіший меседж, менше плутанини → перевірити: візуально в preview + CTR-динаміка 7д після зміни».
- *Приклад:* «Критерій 5 → ad_id=X → `enhanced_media_text` OFF (всередині — testimonial, AI спотворить) → перевірити: preview всіх згенерованих варіантів».
- ⚠️ Будь-яка зміна креативу live = mutate-операція → показати old→new і чекати «так» (правило §9.3 / `safety_rules`). Зміна `add_*`/CTA може **скинути learning** — попередити.

**B. Переробка макета (1 ітерація дизайну).**
- *Angle (крит.1):* «винести angle B у новий ad; у поточному лишити angle A (headline) → тест angles за розділом 5».
- *Readability (крит.3):* «збільшити overlay-текст до X% висоти кадру, контраст до фону; прибрати абзац, лишити 1 тезу».
- *Crop (крит.6):* «перенести CTA/субтитр з нижнього краю в центральну safe-zone; додати окремий 9:16 файл для Reels/Stories».
- *Attention (крит.2):* «замінити стокову заставку на фокусний кадр з контрастом / обличчям; переписати headline під інтригу-суть».

**C. Структурні (паузити / дублювати / тестувати).**
- *Pause:* лише за порогом значущості + performance-причиною (розділ 3).
- *Scale:* дублювати ad/adset з бюджетом тільки якщо Risk LOW–MED **і** CPL≤target **і** n достатній; крок ≤+30% ([[04-budget-scaling/big-spend-scaling]] / [[04-budget-scaling/small-budget]]).
- *Test:* перетворити «message overload» на план тестування angles (розділ 5).

> **Заборонено** видавати фікс без об'єкта й ефекту. «Покращити креатив», «зробити яскравіше», «додати рух» — НЕ фікси. Фікс = {ad_id, що саме змінити, очікуваний ефект, метрика перевірки}.

---

## 5. План тесту (коли причина — angle/overload або потрібен наступний крок)

Будувати ярусами (НЕ міксувати в один тест), порядок незмінний: ⟦[[06-creatives/why-ads-fail]]⟧
1. **Раунд 1 — angles:** виписати причини купити саме для цього ICP; зробити по ОКРЕМОМУ оголошенню на angle (напр. 5 angles → видно 1–2 найсильніші). Який «головний» — вирішує тест, не інтуїція.
2. **Раунд 2 — формат у межах angle-переможця:** image vs video; UGC vs founder-led vs product-demo vs testimonial.
3. **Раунд 3 — хуки/headline/primary text** у межах переможної пари.

Інструмент роздільної доставки і коректного спліту (щоб порівняння було чистим, без перетину аудиторій) → [[12-testing/post-andromeda-testing]]. Зв'язок із сучасною доставкою (чому потрібен ОБСЯГ і РІЗНОМАНІТНІСТЬ кутів, а не варіації) → [[00-philosophy/andromeda-era]].

| ... |

{CPL за 7д/30д vs target; CTR vs медіана adset; breakdowns якщо знімались —

**Правила звіту (з §8/§10 META-PROMPT):**
- Кожна цифра — з джерела `⟦ad_id @період @метрика⟧`. Жодного числа «з пам'яті».
- Принцип Бена цитувати ТІЛЬКИ із застосуванням до цього креативу (не «для краси»).
- Жодного прогнозу без арифметики/механізму («підніме CTR у 2 рази» — заборонено; «overlay OFF прибирає візуальний шум → гіпотеза кращого CTR, міряти 7д» — ок).
- «Не знаю / недостатньо даних» — дозволено й бажано замість вигадки.

## 7. SELF-CHECK (як перевірити / фальсифікувати висновки)

Перед видачею звіту — пройти й включити в нього мінімум 5 пунктів формату «цей висновок хибний, якщо…»:

- [ ] **Джерела розділені.** Кожне твердження про макет → з перегляду креативу / `get_ad_creatives`; кожне число → з `get_insights` із вказаним ad_id+періодом. Жоден принцип Бена не виданий за спостереження по акаунту. *(Хибно, якщо десь «CTR низький» без ⟦ad_id @період⟧ або «overload» без цитати з body.)*
- [ ] **Поріг значущості дотримано.** Жоден performance-вердикт (особливо «паузити») не стоїть на <1000 показів / <3 конв. / <2 днів. *(Хибно, якщо паузу рекомендовано на n=200 показів — тоді правильно: INSUFFICIENT_DATA.)*
- [ ] **Креатив ≠ таргетинг.** Поганий CPL не приписаний автоматично креативу, якщо аудиторія/оффер/LP не зафіксовані. *(Хибно, якщо «креатив поганий» при тому, що adset broad+міксований і чистого тесту не було → потрібен спліт [[12-testing/post-andromeda-testing]].)*
- [ ] **Enhancements звірені з фактом, не з дефолтом.** Стан ON/OFF узятий із `get_ad_creatives`, а не припущений. *(Хибно, якщо написано «animation ON шкодить», а в реальності він OFF.)*
- [ ] **ICP-fit ґрунтований.** Вердикт по критерію 4 спирається на реальний бриф/ICP; за його відсутності — статус NEEDS_ICP, а не вигадане «аудиторія не та». *(Хибно, якщо «не відповідає ЦА» без знання, хто ЦА.)*
- [ ] **Era-валідність.** Жодна порада не суперечить пост-Andromeda підходу (динамічний креатив + enhancements + кроп під плейсменти), не тягне застаріле «строго 1:1, один CTA, текст фіксований». ⟦[[06-creatives/creative-enhancements]]⟧
- [ ] **Кожен фікс дієвий.** Має {object_id + дія old→new + ефект + метрика перевірки}; немає жодного «покращити креатив» без конкретики.
- [ ] **Mutate-гейт.** Будь-яка пропозиція змінити/паузити live-креатив подана як план old→new із запитом підтвердження; враховано можливий reset learning.

**Як фальсифікувати весь аудит (зовнішня перевірка):** взяти один фікс класу A (enhancement-toggle), застосувати на ОДНОМУ ad, лишити двійника без зміни, виміряти CTR/CPL за 7 днів при n≥1000 — якщо різниці немає, гіпотеза цього критерію для даного креативу не підтвердилась (і це нормальний результат, а не «провал аудиту»).
