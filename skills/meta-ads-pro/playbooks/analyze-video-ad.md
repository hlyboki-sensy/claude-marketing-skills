# Аудит відео-оголошення (video ad)

> **Призначення.** Дати по ОДНОМУ відео-оголошенню дієвий вердикт: **Risk Score (0–100)** + чек-ліст по 7 критеріях (hook 0–3 с · thumbstop/hold · субтитри · pacing · один angle · CTA · формат під кроп) + конкретні фікси та **план тесту хуків через Creative Testing tool**. Не «зробити кращий хук», а: «hook 0–3 с слабкий (hook rate 18% за ⟦ad_id @last_7d⟧ проти медіани адсета 31%) → запустити hook-тест: 1 body × 6 нових хуків через Creative Testing tool, key metric = cost per lead; паузити ad_id=X (CPL 3.4× target за 7д, n=5 конв.)».
>
> **Hook вирішує все.** Перші ~3 секунди — найдорожча частина відео: той самий ролик з іншим хуком дає кардинально інший результат. ⟦[[12-testing/post-andromeda-testing]]⟧ Тому критерій 1 (hook) має найбільшу вагу, а основний клас фіксів — тест хуків, а не «переробити все відео».
>
> **Два режими.** (А) **Pre-flight** — відео ще не запущене, даних немає → аудит лише по самому ролику/правилах (Risk Score без performance-блоку, video-метрик немає). (Б) **Live** — оголошення крутиться → той самий чек-ліст + performance-діагностика з video-метрик `get_insights` (hook rate / hold / thruplay / CPL), era-валідна, з порогами значущості.
>
> **Залізне правило.** KNOWLEDGE (принцип з `knowledge/`) ≠ DATA (цифра з Marketing API). Принцип каже «hook = перші 3 секунди, тестуй хуки»; чи СЛАБКИЙ хук у КОНКРЕТНОМУ відео — встановлюється з video-метрик (`video_p25_watched`, `video_thruplay`, CTR) і перегляду ролика, а не з голови. Кожна цифра у звіті — з джерела (ad_id / період / метрика).

## 0. Коли застосовувати

- Користувач дав 1 відео (файл/посилання або ad_id) і просить «оціни / що не так / чому погано тримає / before-launch перевірка відео».
- Перед масштабуванням відео-зв'язки (дублювання з бюджетом) — швидкий risk-gate саме по hook/hold.
- У межах `audit-account` — як дрилл-даун на «найгірше за CPL відео» / «single point of failure» video-ad, або коли акаунт «тестує хуки» неправильно (звичайними адсетами).
- Коли є гіпотеза «відео не тримає увагу» — щоб відділити проблему **хука (0–3 с)** від проблеми **утримання тіла (3 с → кінець)** від проблеми **аудиторії/оферу**.

Якщо відео багато — застосовувати цей playbook **по одному**; зведення (creative concentration, який % бюджету на TOP-1 відео) робити окремо, як у `audit-account`.

> **Відмінність від `analyze-creative.md`.** Той playbook — про статику (image ad). Цей — про відео: додаються осі **часу** (hook 0–3 с, hold/retention-крива, pacing) і **звуку** (субтитри, бо більшість дивиться без звуку). Спільні критерії (один angle, ICP, кроп) трактуються через video-специфіку. Для UGC/founder-led/testimonial вибору формату — це наступний раунд після angle, див. розділ 6.

## 1. Збір даних (read-before-write)

### 1.1 Що завжди читаємо (обидва режими)

```
get_ads(adset_id=...)                         # знайти ad_id, status, name, чи це video creative
get_ad_creatives(ad_id=...)                   # object_story_spec.video_data: video_id, title (headline),
                                              #   message (primary text), call_to_action_type,
                                              #   image (thumbnail/cover), captions/субтитри (якщо є),
                                              #   degrees_of_freedom_spec / creative_features_spec
                                              #     (які enhancements ON/OFF — для video менше опцій)
```

- **Обов'язково подивитись сам ролик очима** (відкрити файл/посилання). Без перегляду відео критерії 1–4 (hook, thumbstop, субтитри, pacing) **не оцінюються** — так і сказати: «потрібен сам відеофайл/посилання, інакше п.1–п.4 чек-листа не оцінюються, лише п.5–п.7 з метаданих».
- Подивитись **з вимкненим звуком** хоча б один прохід — це і є тест критерію 3 (субтитри): якщо без звуку незрозуміло, що відбувається, — це прапор, бо більшість аудиторії дивиться саме так. ⟦[[06-creatives/creative-enhancements]]⟧
- З `get_ad_creatives` витягнути блок **enhancements** (у Marketing API `degrees_of_freedom_spec` / `creative_features_spec`): для відео опцій менше, ключове — `add_video_effects` (color/visual sharpening, дефолт OFF), `visual_touch_ups`/`image_uncrop` (кроп під плейсменти), `text_improvements` (перестановка primary/headline), `music`. Це критично для критеріїв 6–7.
- Зафіксувати **тривалість відео** і де стоїть **CTA** (вшитий у відео в кінці / тільки кнопка call_to_action_type / і там, і там).

### 1.2 Що додатково читаємо в режимі (Б) Live — video-метрики

```
get_insights(object_id=ad_id, level="ad", time_range="last_7d",
  fields="impressions,clicks,ctr,spend,actions,video_play_actions,
          video_thruplay_watched_actions,
          video_p25_watched_actions,video_p50_watched_actions,
          video_p75_watched_actions,video_p100_watched_actions,
          video_avg_time_watched_actions")
get_insights(object_id=ad_id, level="ad", time_range="last_30d", fields=...той самий набір...)

# опційно — перевірити гіпотезу «кроп/плейсмент б'є по hold»:
get_insights(object_id=ad_id, level="ad", time_range="last_30d",
             breakdowns="publisher_platform")   # FB feed / IG feed / Reels / Stories / AN
# опційно — ICP-fit за фактом:
get_insights(object_id=ad_id, level="ad", time_range="last_30d", breakdowns="age")  # + окремо gender
```

> ⚠️ **Назви полів — звірити з реальною відповіддю Marketing API.** Точний набір video-полів (`video_play_actions`, `video_thruplay_watched_actions`, `video_pXX_watched_actions`, `video_avg_time_watched_actions`) залежить від версії Marketing API і того, що повертає конкретний Meta Marketing API. Якщо інструмент не віддає якесь поле — не вигадувати число, а зафіксувати в data-gaps: «hold rate не порахований, бо немає `video_p50_watched_actions` у відповіді — потрібен інший fields-запит або UI-вигляд». Не плутати знання (як рахується hook rate) з даними (що реально повернулось).

### 1.3 Похідні video-метрики (формули — рахувати лише коли є поля)

Усе нижче — це **визначення метрик** (KNOWLEDGE), які наповнюються тільки цифрами з Marketing API (DATA). Глосарій:

| Метрика | Формула | Що означає | Поріг «дивитись» |
|---|---|---|---|
| **Hook rate** (3-sec / thumbstop) | `video_3sec_views / impressions` (або `video_play_actions[3s] / impressions`) | яка частка показів зачепилась перші ~3 с — **прямий проксі сили хука** | низький відносно медіани адсета → слабкий hook |
| **Hold rate** (retention) | `video_p100_watched / video_3sec_views` (доходження до кінця від тих, хто стартував) | чи тіло відео утримує після хука | різке падіння на p25→p50 = слабке тіло/pacing |
| **ThruPlay rate** | `video_thruplay_watched / impressions` | %, що додивились до ThruPlay (15 с або кінець, якщо коротше) | агрегований сигнал «хук + утримання» |
| **Retention-крива** | p25 / p50 / p75 / p100 (кожен / 3sec або / play) | ДЕ саме відвалюються глядачі | обрив на p25 → проблема хука/перших секунд; рівне сповзання → pacing |
| **Avg watch time** | `video_avg_time_watched` | середній час перегляду в секундах | порівнювати з тривалістю ролика |
| **CTR (link)** | `clicks / impressions` | чи перетворюється увага на клік | низький при норм. hold → слабкий CTA/оффер |
| **CPL** | `spend / leads` (leads з `actions` під ваш event) | кінцева ефективність | vs `target_CPL` з брифу |

> **Hook rate vs hold — це дві різні діагнози.** Низький **hook rate** при нормальному hold → проблема перших 3 секунд (фікс = тест хуків). Нормальний hook rate, але різкий обрив на **p25→p50** → проблема тіла/pacing (фікс = переробити тіло, не хук). Не змішувати: «hook rate» і «hold rate» дають різні рекомендації, тому в звіті розділяти.

**Поріг значущості (з `safety_rules` + §9 META-PROMPT):** не виносити performance-вердикт, якщо **<1000 показів** АБО **<3 конверсій** АБО **<2 днів** відкрутки. Менше — статус `INSUFFICIENT_DATA`: «потрібно ще покази/конверсії, не змінювати». Для video-метрик окремо: hook rate/hold рахувати лише при ≥1000 показів; на breakdown-зрізі з <300 показів — це гіпотеза, не вердикт «Reels гірші за Feed».

> ⚠️ **Не плутати відео з таргетингом/оффером.** Поганий CPL ще не означає погане відео. Чесна video-діагностика: high CPL **при низькому hook rate** → справді хук/відео; high CPL **при нормальних hook+hold** → причина деінде (аудиторія/оффер/LP), відео не винне. Якщо адсет міксований (різні аудиторії/багато відео) — «CPL міксує відео+аудиторію, для чистого тесту відео потрібен роздільний спліт → [[12-testing/post-andromeda-testing]]».

## 2. Сім критеріїв аудиту (ядро)

Кожен критерій: **що перевіряємо → джерело сигналу (KNOWLEDGE vs DATA) → як оцінюємо → бали ризику**. Бали підсумовуються в Risk Score (розділ 3). Принципи — з `knowledge/`; факт «слабко/сильно» — з самого ролика і з video-метрик.

### Критерій 1 — Hook 0–3 секунди (вирішальний) · вага до 30

**Принцип [P]:** **хук = перші ~3 секунди відео** — вирішальний; той самий ролик з іншим хуком дає кардинально інший результат. Дрібна зміна на старті може дати велику різницю в перформансі. ⟦[[12-testing/post-andromeda-testing]]⟧ Велика частина задачі креативу — привернути увагу в перші моменти. ⟦[[06-creatives/creative-enhancements]]⟧

**Сигнал — перегляд перших 3 с ролика (KNOWLEDGE-чек):**
- Хук дає миттєву причину НЕ скролити далі (питання / несподіванка / сильне твердження / рух / впізнаваний біль аватара) у перші ~3 с?
- Хук говорить мовою аватара і б'є в один біль/бажання (а не «привіт, мене звати…» / повільний інтро / лого-заставка на весь старт)?
- Найважливіше НЕ відкладене на 5–10 с (за цей час глядач уже відвалився)?

**Сигнал — DATA (режим Б, n≥1000):** **hook rate (3-sec/thumbstop)** проти медіани адсета. Низький hook rate при нормальному CPM = слабкий старт, а не доставка. Якщо retention-крива має різкий обрив уже на **p25** — підтвердження, що проблема саме в перших секундах.

| Стан хука | Бали |
|---|---:|
| Сильний hook: миттєвий гачок у перші ~3 с, мова аватара; DATA (якщо є) — hook rate ≥ медіани адсета | 0 |
| Хук робочий, але «розгін» 1–2 с зайвий / гачок м'який | +12 |
| Слабкий старт: повільне інтро / лого-заставка / «привіт, я…» перші секунди; DATA — hook rate помітно < медіани | +22 |
| Хука фактично немає: перші 3 с нічого не обіцяють, обрив на p25; high CPL при низькому hook rate | +30 |

**Фікс — не «зроби кращий хук» абстрактно, а тест хуків:** «той самий body × кілька нових хуків через Creative Testing tool» (розділ 6). Який хук переможе — вирішує тест, не інтуїція. ⟦[[12-testing/post-andromeda-testing]]⟧ Це найдешевший важіль: міняється лише старт, тіло лишається.

### Критерій 2 — Thumbstop / hold rate (утримання) · вага до 20

**Принцип [P]:** хук зупиняє скрол, але тіло має **утримати**; це окрема від хука задача. Реалістична оцінка — по retention-кривій: де саме відвалюються глядачі. *(Утримання як окрема вісь від «зупинки скролу» випливає з того, що hook = перші 3 с, а далі результат залежить від решти ролика. ⟦[[12-testing/post-andromeda-testing]]⟧)*

**Сигнал — DATA (режим Б, головний для цього критерію):** **hold rate** (`p100 / 3sec`) і форма retention-кривої p25→p50→p75→p100; **ThruPlay rate**; **avg watch time** vs тривалість.

**Сигнал — KNOWLEDGE/перегляд (pre-flight, коли даних немає):** оцінити ризик провисання на око — чи є «мертві» секунди, затягнутий сегмент, момент, де хочеться скролити.

Діагностика за кривою (режим Б, n≥1000):

| Форма retention-кривої | Діагноз | Бали |
|---|---|---:|
| Плавне сповзання, високий ThruPlay (≥ медіани адсета) | тіло тримає | 0 |
| Помірний обрив на p50–p75 | провисання в середині (pacing) | +10 |
| Різкий обрив на p25 одразу після хука | хук зачепив не тих / обіцянка хука не підхоплена тілом | +15 *(зв'язати з крит.1 і крит.4)* |
| Дуже низький hold/ThruPlay по всій кривій | тіло не утримує / не той оффер | +20 |

> **Hook rate високий, але hold низький** — типовий патерн «клікбейтний хук, слабке тіло»: хук зачепив, але обіцянку не виконано. Фікс — НЕ ще агресивніший хук (погіршить), а переробка тіла / вирівнювання хук↔тіло. Якщо ж hold нормальний, а проблема лише на p25 — фікс у хуці (крит.1).

### Критерій 3 — Субтитри (дивляться без звуку) · вага до 15

**Принцип [P]:** **багато людей дивляться без звуку** — music у Meta це «невеликий важіль» саме тому. ⟦[[06-creatives/creative-enhancements]]⟧ Отже сенс відео має передаватись **візуально + субтитрами**, не покладаючись на аудіо.

**Сигнал — перегляд ролика з ВИМКНЕНИМ звуком** (KNOWLEDGE-чек):

Чек (кожне «ні» додає бали):
- Є субтитри/burned-in текст, що передають ключовий меседж без звуку? → ні: +7
- Хук (перші 3 с) зрозумілий без звуку (текст/візуал, а не лише озвучка)? → ні: +5 *(зв'язано з крит.1 — німий хук особливо критичний)*
- Субтитри читабельні (розмір, контраст, не залітають надто швидко, синхронні)? → ні: +3

> **Зчеплення з кропом (крит.7).** Субтитри мають бути **по центру**, не біля нижнього краю: Meta кропить/розтягує під 4:5 та інші формати, нижні субтитри при кропі зникають. ⟦[[06-creatives/creative-enhancements]]⟧ Якщо субтитри внизу — це одночасно прапор крит.3 (ризик зрізу) і крит.7 (safe-zone). Не дублювати бали — основний штраф ставити в крит.7, у крит.3 лише відзначити.

**DATA-нюанс (режим Б):** прямого «sound-off rate» в `get_insights` зазвичай немає — **не вигадувати** частку «X% без звуку». Це знання-аргумент (більшість без звуку), а не вимірювання по акаунту; так і подавати.

### Критерій 4 — Pacing і один angle на відео · вага до 20

**Принцип [P]:** **один ad angle на одне оголошення** — не зсипати всі причини купити в одне відео; надлишок інформації вбиває оголошення, проспект не встигає зрозуміти «що це, що робить, чи мені підходить». Автор бачить НАБАГАТО більше задто складних оголошень, ніж задто простих. ⟦[[06-creatives/why-ads-fail]]⟧

**Сигнал — перегляд ролика + `get_ad_creatives` (message/title):** порахувати, скільки різних причин купити / меседжів намагається покрити відео, і чи рівний темп.

Чек:
- **Один наскрізний angle** (хук, тіло, CTA — про одне), без message overload? → ні (2–3 angle злиті): +10 *(в одне відео впхнуто кілька офферів/причин)*
- **Pacing** рівний, без затягнутих «мертвих» секунд і без каші монтажу? → ні: +6
- Тривалість адекватна меседжу (не розтягнуто заради тривалості; ключове — на початку)? → ні: +4

**Прапор «message overload»** піднімати, якщо відео намагається сказати все одразу. Фікс — не «спростити» абстрактно, а: «винести angle B і angle C в ОКРЕМІ відео; у цьому лишити angle A». Який angle лишати — вирішує тест angles, не інтуїція ⟦[[06-creatives/why-ads-fail]]⟧ (порядок раундів — розділ 6). Pacing-провал, що збігається з обривом retention на конкретному p-кроці (крит.2), — підтверджує, де саме переробляти тіло.

### Критерій 5 — CTA (заклик до дії) · вага до 10

**Принцип [P]:** оголошення має дати людині швидко зрозуміти «що це, чому купити, ось для кого» і **що робити далі** — дисципліноване сфокусоване оголошення з одним меседжем. ⟦[[06-creatives/why-ads-fail]]⟧ CTA — точка переходу уваги в дію.

**Сигнал — перегляд кінцівки ролика + `get_ad_creatives` (`call_to_action_type`):**

Чек (кожне «ні» додає бали):
- Є ОДИН чіткий CTA (вшитий у відео та/або кнопка), що каже, що робити далі? → ні: +5
- CTA узгоджений з angle/оффером (не «купи зараз» там, де меседж був про безкоштовний урок)? → ні: +3
- `call_to_action_type` (кнопка) виставлений і доречний для objective (Learn More / Sign Up / Get Offer тощо)? → ні: +2

**DATA-сигнал (режим Б):** нормальні hook+hold, але низький **CTR** → слабкий/нечіткий CTA або розрив «обіцянка хука ↔ дія». Якщо при цьому CTR падає лише на одному плейсменті (breakdowns=publisher_platform) — гіпотеза, що кнопка/кінцівка не лягає під формат.

> Enhanced CTA (цикл варіантів CTA) — для **image** Ben радить пробувати; для **video** опцій менше і кнопка простіша — не нав'язувати, перевірити, що тип кнопки взагалі коректний.

### Критерій 6 — Creative enhancements для відео · вага до 10

**Принцип [P]:** рекламодавець — **фільтр**: вмикати лише enhancement-и, що пасують ЦЬОМУ креативу. Для **video опцій менше, ніж для image** (відео вже рухається/має звук, оверлеї важче зробити коректно). Ключове для відео: **`add_video_effects`** (color/visual sharpening) — дефолт **OFF**: лишати OFF, якщо відео вже яскраве/контрастне навмисно; вмикати, якщо тьмяне. Music обов'язково перевіряти руками (дефолт буває «super corporate»), але це невеликий важіль (більшість без звуку). ⟦[[06-creatives/creative-enhancements]]⟧

**Сигнал — блок enhancements з `get_ad_creatives` (ON/OFF) + характер ролика.**

Звірити по матриці (кожне порушення додає бали; макс +10):

| Enhancement (video) | Дефолт | Коли ON | Коли OFF / перевірити | Порушення → бали |
|---|---|---|---|---:|
| `add_video_effects` (color/sharpening) | OFF | відео тьмяне/блякле | відео вже яскраве навмисно (ефект «переб'є») | +3 |
| `music` | ON | додатковий гачок для тих, хто зі звуком | дефолтний «corporate» трек не пасує vibe → перевірити/OFF | +2 *(малий важіль — не переоптимізовувати)* |
| `text_improvements` (перестановка primary/headline) | ON | майже завжди лишати ON (Meta підбирає під людину+плейсмент) | рідкісні крайні випадки | 0 (лишати ON) |
| `visual_touch_ups` / `image_uncrop` (кроп під плейсменти) | ON | майже завжди ON (гнучкість під формати) | — (керується критерієм 7 — safe-zone) | 0 (лишати ON, але див. крит.7) |

> Для відео НЕ застосовні image-only `add_overlays` / `add_animation` / `enhanced_media_text` / AI image variations — не звітувати про них як про проблему відео. Якщо Marketing API показав їх у video-creative — це або image-частина asset_feed, або шум; зазначити, не штрафувати помилково. ⟦[[06-creatives/creative-enhancements]]⟧

### Критерій 7 — Формат під кроп (safe-zone, важливе по центру) · вага до 10

**Принцип [P]:** Meta кропить/розтягує креатив під 4:5 та інші формати й плейсменти (visual touch-ups / flex media / uncrop), тому **проєктуй під це**: важлива інфо/субтитри — по ЦЕНТРУ, не скраю (не вгорі/внизу), інакше при кропі зникне. Розподіл показів: vertical (Reels/Stories) і square (feeds) — багато, horizontal — мало. ⟦[[06-creatives/creative-enhancements]]⟧

**Сигнал — перегляд ролика (де лежать субтитри/ключова інфо/CTA) + які aspect-ratio файли завантажені (`get_ad_creatives` / asset_feed).**

Чек (кожне «ні» додає бали):
- Субтитри / ключова інфо / CTA — у центральній safe-zone (не у верхніх/нижніх ~15%, не впритул до країв)? → ні: +5
- Відео вертикальне або є окрема vertical-версія (9:16) під Reels/Stories — або свідомо покладаємось на uncrop? → ні: +3
- Ролик не «ламається» при кропі в 4:5 (нічого важливого/обличчя/тексту не відрізає)? → ні: +2

**DATA-підтвердження (режим Б):** `get_insights breakdowns=publisher_platform` — якщо на vertical-плейсментах (Reels/Stories) hold/CTR/CPL помітно гірші при n≥1000 на зрізі → гіпотеза «відео не адаптоване під vertical-кроп / субтитри зрізає». При малому n — лише гіпотеза, не вердикт.

## 3. Risk Score (0–100) і вердикт

```
Risk Score = sum(бали критеріїв 1..7)
           = hook(≤30) + hold(≤20) + subtitles(≤15) + pacing_angle(≤20)
           + cta(≤10) + enhancements(≤10) + crop(≤10)
```
> Максимум сирих ваг = 115; за наявності рідкісних накладок капувати на **100**. Вага хука (30) і hold (20) свідомо найбільші — це осі, що найсильніше рухають результат відео.

### Класи ризику та дія

| Risk Score | Клас | Іконка | Вердикт і дія |
|---|---|---|---|
| 0–25 | LOW | 🟢 | Готове до запуску / до масштабування. Дрібні фікси (зазвичай enhancements-toggle, дрібне subtitle/CTA). |
| 26–50 | MEDIUM | 🟡 | Запускати/лишати можна, але з пріоритетними фіксами; для масштабу — спершу полагодити hook/subtitles. |
| 51–75 | HIGH | 🟠 | Не масштабувати. Запустити **тест хуків** (розділ 6) і переробити топ-2 критерії перед (пере)запуском. У live — кандидат на заміну. |
| 76–100 | CRITICAL | 🔴 | Не запускати / у live — паузити (за умови порогу значущості). Відео переробляти: новий hook + ревізія тіла; тест angles+хуків з нуля. |

### Накладання з performance-вердиктом (тільки режим Б, n достатній)

Risk Score (якість ролика) і performance (CPL/CTR/hold) — **різні осі**. Зводимо через **hook rate**, щоб не плутати «слабкий хук» зі «слабким оффером/аудиторією»:

| | Risk Score LOW–MED | Risk Score HIGH–CRIT |
|---|---|---|
| **CPL ≤ target** | 🟢 Лишити; кандидат на дублювання з бюджетом *(крок ≤+30%, [[05-optimization/learning-phase]]: не частіше 1 зміни/7 днів)* | 🟡 Працює попри слабкий ролик — лишити, але підготувати чистіший хук (ризик швидкого вигоряння) |
| **CPL > 3× target, низький hook rate** (n≥3 конв., ≥1000 пок.) | 🟠 Ролик ок, але старт слабкий → перш ніж убивати: **тест хуків**; якщо й кращий хук не рятує — причина деінде | 🔴 **Паузити ad_id=X** (CPL Nх target за 7д, n=K конв., hook rate M%) + тест хуків + ревізія тіла |
| **CPL > 3× target, hook+hold нормальні** | 🟠 Відео НЕ винне — перевірити аудиторію/оффер/LP, не вбивати ролик наосліп | 🟡 Подвійна проблема: і ролик, і ймовірно оффер/таргет — розділити спліт-тестом [[12-testing/post-andromeda-testing]] |

> **Anti-hallucination guard.** Якщо n<1000 показів або <3 конв. — performance-клітинку НЕ заповнювати: «INSUFFICIENT_DATA, рішення про паузу передчасне, чекати». Вердикт «паузити» з performance-причини **вимагає** ad_id + період + CPL + кількість конверсій (+ hook rate, якщо твердиш про хук). Без цього паузу не рекомендувати (правило ad-eater з `safety_rules`: CPL>3× target → пауза, але після порогу значущості). «Низький hook rate» без `⟦ad_id @період⟧` — заборонено.

## 4. Фікси (дієві, з прив'язкою до об'єкта)

Кожен фікс — конкретна дія над конкретним об'єктом. Формат: **{критерій} → {object_id} → {дія} → {очікуваний ефект + механізм} → {як перевірити}**. Категорії за «вартістю» правки:

**A. Enhancements-toggle (миттєво, без перероблення ролика).** Найдешевший клас — змінюється на рівні ad creative, новий файл не потрібен.
- *Приклад:* «Крит.6 → ad_id=X → `add_video_effects` OFF→ON (відео тьмяне, sharpening підніме контраст/увагу) ⟦[[06-creatives/creative-enhancements]]⟧ → перевірити: preview + hook rate/CTR 7д».
- *Приклад:* «Крит.6 → ad_id=X → `music` — відкрити customize, перевірити дефолтний трек; якщо "corporate" і не пасує → OFF/замінити (малий важіль, не пріоритет)».

**B. Правка без повного перезняття (субтитри / CTA / safe-zone).**
- *Субтитри (крит.3):* «додати burned-in субтитри на весь ролик; зробити хук зрозумілим без звуку».
- *Safe-zone (крит.7):* «перенести субтитри з нижнього краю в центральну зону; додати окрему 9:16 версію під Reels/Stories».
- *CTA (крит.5):* «додати/посилити вшитий CTA у кінці + виставити коректний `call_to_action_type`».

**C. Перезняття / новий монтаж (hook / тіло / pacing / angle).**
- *Hook (крит.1):* НЕ «зроби кращий хук» — а «зняти/змонтувати кілька нових хуків (0–3 с) до того ж body → тест хуків (розділ 6)».
- *Тіло/pacing (крит.2, крит.4):* «вирізати провисання на p50 (де обрив retention за ⟦ad_id @last_30d⟧); вирівняти обіцянку хука з тілом».
- *Angle (крит.4):* «винести angle B у нове відео; у поточному лишити angle A → тест angles (розділ 6)».

**D. Структурні (паузити / дублювати / тестувати).**
- *Pause:* лише за порогом значущості + performance-причиною (розділ 3); у live = mutate, потрібне підтвердження.
- *Scale:* дублювати відео/адсет з бюджетом тільки якщо Risk LOW–MED **і** CPL≤target **і** n достатній; крок ≤+30%; пам'ятати, що зміна рестартить learning phase ⟦[[05-optimization/learning-phase]]⟧.
- *Test:* перетворити «слабкий hook» / «message overload» на план тестування хуків/angles (розділ 6).

> ⚠️ **Mutate-гейт.** Будь-яка зміна live-оголошення (toggle enhancement, заміна креативу, пауза, +бюджет) = mutate-операція → показати old→new, обґрунтування цифрами, очікуваний ефект, ризик і **чекати «так»** (§9.3 META-PROMPT / `safety_rules`). Зміна креативу чи додавання нових оголошень **скидає learning phase** — попередити; не вносити зміни в адсет частіше ніж раз на 7 днів і заливати нові хуки **однією партією**, а не по одному щодня. ⟦[[05-optimization/learning-phase]]⟧

> **Заборонено** видавати фікс без об'єкта й ефекту. «Зробити кращий хук», «динамічніше», «додати енергії» — НЕ фікси. Фікс = {ad_id, що саме змінити, очікуваний ефект+механізм, метрика перевірки}.

## 5. План тесту ХУКІВ (центральний для відео) + ярус angles

Це головний результат playbook для відео, бо hook вирішує найбільше, а тестувати хуки **можна і треба** — через правильний інструмент. ⟦[[12-testing/post-andromeda-testing]]⟧

### 5.1 Чому НЕ звичайними адсетами

Класичне A/B «кинути кілька схожих відео (різні хуки, те саме тіло) в один адсет і чесно порівняти» **зламане**: Meta схлопує схожі креативи й крутить лише одне. ⟦[[12-testing/post-andromeda-testing]]⟧ Тому хуки (схожі варіації одного ролика) — **тільки через Creative Testing tool** (роздільна доставка). Якщо клієнт «тестує хуки» звичайними адсетами — це прапор в аудиті.

### 5.2 Hook method (як налаштувати)

- **Структура:** взяти 1–2 «тіла» (body) відео × кілька хуків. Канонічний приклад: **2 body × 10 хуків = 20 оголошень**; для малих бюджетів — менше (напр. 1 body × 4–6 хуків). ⟦[[12-testing/post-andromeda-testing]]⟧
- **Де:** Ads Manager → рівень **оголошення** → блок **Creative testing** (Creative Testing tool). Саме він тримає доставку роздільно для схожих варіацій.
- **Параметри тесту:** 2–5 версій · % бюджету на тест · тривалість (дефолт **7 днів**, збільшити за малого обсягу конверсій) · **key metric = cost per lead / sale, а НЕ cost per post engagement**. ⟦[[12-testing/post-andromeda-testing]]⟧
- **Після тесту:** Meta зазвичай переливає бюджет на переможця (перевірити вручну — не завжди); переможний хук поставити в адсет **поруч з ІНШИМИ, дуже різними оголошеннями** (а не з його ж варіаціями); програші вимкнути. ⟦[[12-testing/post-andromeda-testing]]⟧
- **Значущість:** переконатися, що тестовий період дав достатньо **конверсій** (а не лише показів/engagement) для висновку; інакше продовжити, не «вгадувати» переможця.

> **Дуже різні оголошення** (image vs video, інший візуал/кут) інструмент не потребує — Meta й так їх розрізнить; Creative Testing tool потрібен саме для **схожих** (різні хуки одного ролика, прайс-банер, B&W vs color). ⟦[[12-testing/post-andromeda-testing]]⟧

### 5.3 Ярус angles (коли проблема — message overload або вибір формату)

Порядок раундів незмінний (НЕ міксувати в один тест): ⟦[[06-creatives/why-ads-fail]]⟧
1. **Раунд 1 — angles:** виписати причини купити саме для цього ICP; по ОКРЕМОМУ оголошенню на angle (напр. 5 angles → 1–2 найсильніші). Який головний — вирішує тест.
2. **Раунд 2 — формат у межах angle-переможця:** image vs video; UGC vs founder-led vs product-demo vs testimonial.
3. **Раунд 3 — хуки** (video) / headline / primary text у межах переможної пари → знову через Creative Testing tool (5.2).

> Тобто **тест хуків — це раунд 3**, він має сенс на вже доведеному angle+форматі. Якщо angle ще не валідований — спершу тест angles, інакше відполіруєш хук для слабкого меседжу. Зв'язок із сучасною доставкою (чому потрібні ОБСЯГ і РІЗНОМАНІТНІСТЬ кутів, а не 20 варіацій-близнюків) → [[00-philosophy/andromeda-era]].

- «Не знаю / недостатньо даних» — дозволено й бажано замість вигадки (немає video-полів → так і написати, не вигадувати retention).

## 7. SELF-CHECK (як перевірити / фальсифікувати висновки)

Перед видачею звіту — пройти й включити в нього мінімум 5 пунктів формату «цей висновок хибний, якщо…»:

- [ ] **Джерела розділені (KNOWLEDGE ≠ DATA).** Кожне твердження про ролик → з перегляду відео / `get_ad_creatives`; кожне число (hook rate, hold, CPL) → з `get_insights` із вказаним ad_id+періодом. Жоден принцип Бена («хук вирішує», «один angle») не виданий за спостереження по акаунту. *(Хибно, якщо десь «hook rate низький» без ⟦ad_id @період⟧, або «message overload» без перегляду/цитати з message.)*
- [ ] **Hook ≠ hold розділено.** Діагноз указує, проблема в перших 3 с (hook rate) чи в тілі (обрив retention на p25→p50). Фікс відповідає діагнозу (тест хуків vs переробка тіла), а не «покращити відео» загалом. *(Хибно, якщо радимо переробити хук, тоді як retention-крива показує обрив на p75 — там проблема pacing, не хука.)*
- [ ] **Поріг значущості дотримано.** Жоден performance-вердикт (особливо «паузити») не стоїть на <1000 показів / <3 конв. / <2 днів. Hook rate/hold не пораховані на <1000 показів. *(Хибно, якщо паузу рекомендовано на n=200 показів — тоді правильно: INSUFFICIENT_DATA.)*
- [ ] **Відео ≠ таргетинг/оффер.** Поганий CPL не приписаний автоматично відео, якщо hook+hold нормальні й аудиторія/оффер/LP не зафіксовані. *(Хибно, якщо «відео погане» при нормальному hook rate і міксованому broad-адсеті без чистого тесту → потрібен спліт [[12-testing/post-andromeda-testing]].)*
- [ ] **Метрики реальні, не припущені.** Video-поля (`video_pXX_watched`, `video_thruplay`, `video_play_actions`) узяті з фактичної відповіді Marketing API; яких немає — у data-gaps, а не «приблизно». «Sound-off rate» не вигаданий числом. *(Хибно, якщо наведено «hold 42%», а поле p100 у відповіді відсутнє.)*
- [ ] **Enhancements звірені з фактом, не з дефолтом, і саме video-набір.** Стан ON/OFF — з `get_ad_creatives`; image-only опції (add_overlays/animation/AI-variations) не приписані відео як проблема. *(Хибно, якщо «animation шкодить відео» — це image-enhancement, для video не той набір.)*
- [ ] **Тест хуків — через правильний інструмент і метрику.** План каже Creative Testing tool (не звичайні адсети для схожих варіацій) і key metric = cost per lead/sale (не post engagement). *(Хибно, якщо радимо «закинути 5 відео з різними хуками в один адсет» — Meta схлопне.)*
- [ ] **Ярус дотримано.** Тест хуків запропоновано на вже валідованому angle/форматі; якщо angle не доведений — спершу тест angles. *(Хибно, якщо поліруємо хук для меседжу, що сам не пройшов angle-тест.)*
- [ ] **Era-валідність.** Жодна порада не тягне застаріле («20 відео-варіацій-близнюків», «строго 9:16», «один CTA назавжди як єдина істина») всупереч пост-Andromeda (обсяг РІЗНИХ кутів + кроп під плейсменти + динамічний креатив). ⟦[[00-philosophy/andromeda-era]]⟧
- [ ] **Mutate-гейт + learning.** Будь-яка зміна live-відео подана як план old→new із запитом підтвердження; враховано, що заміна креативу/нові хуки **скидають learning phase**, і не частіше 1 зміни/7 днів. ⟦[[05-optimization/learning-phase]]⟧
- [ ] **Кожен фікс дієвий.** Має {object_id + дія old→new + ефект/механізм + метрика перевірки}; немає жодного «зробити кращий хук» без конкретики.

**Як фальсифікувати весь аудит (зовнішня перевірка):** запустити **hook-тест** через Creative Testing tool (той самий body × нові хуки, key metric = cost per lead, ≥7 днів, достатньо конверсій). Якщо новий хук НЕ покращив cost per lead / hook rate проти оригіналу при достатньому n — гіпотеза «проблема в хуці» для цього відео не підтвердилась (нормальний результат, а не «провал аудиту»; шукати причину в тілі/оффері/аудиторії). Аналогічно для крит.3: додати субтитри на одну версію, лишити двійника без них, порівняти hold/CTR за 7 днів при n≥1000.
