# Ad-Account Safety + Анти-галюцинації

> **Що це.** Єдиний операційний документ, що зводить §9 META-PROMPT (анти-галюцинації) і `ai-ads-agent/config/knowledge/safety_rules.md` (ліміти/пороги/історія) в один контракт. Усе, що змінює живий кабінет, проходить через ці правила. Якщо правило і запит користувача суперечать — **спитати, не вгадувати**.
>
> **Сфера дії.** Аналіз (read-only) і оптимізація (mutate) через Meta Marketing API. Плейсхолдери: акаунт `act_XXX`, об'єкти — `campaign_id` / `adset_id` / `ad_id`. Цільові метрики (`target_CPL`, `target_ROAS`) беруться **з брифу акаунта**, а не вигадуються.
>
> **Золоте правило.** Кожна цифра про кабінет — прочитана з Marketing API цієї сесії. Кожен принцип — з `knowledge/` з посиланням `[[category/subcategory]]`. Ці два джерела **ніколи не змішуються**.

## 0. Дві сутності, які не можна плутати: KNOWLEDGE vs DATA

| | **KNOWLEDGE** (принцип) | **DATA** (акаунт) |
|---|---|---|
| Джерело | `knowledge/**` (методологія Ben Heath) | Meta Marketing API цієї сесії |
| Приклад | «пост-Andromeda адсет норм тримати 20+ різнопланових креативів» [[00-philosophy/andromeda-era]] | «adset_id=A має 4 активні ad, get_adsets @2026-06-03» |
| Як цитувати | `[[00-philosophy/andromeda-era]]` | `(get_insights adset_id=A, last_7d, @2026-06-03: CPL $X)` |
| Що НЕ можна | видати принцип за спостереження | видати спостереження за універсальний закон |

**Заборонена фраза:** «Бен каже, що в тебе вигоряння аудиторії». Принцип Бена описує загальний патерн; **виявити патерн у конкретному акаунті** можна тільки з DATA (frequency, тренд CTR за датами). Правильно: «Принцип: frequency >2 → ризик виснаження [[06-creatives/why-ads-fail]]. У DATA: adset_id=A frequency_7d=3.1 (get_insights @2026-06-03) → патерн підтверджується».

## 1. Read-before-write (анти-галюцинація №1)

**Жодного mutate-виклику без свіжого read цього ж об'єкта в цій сесії.**

Перед будь-якою зміною об'єкта прочитати його поточний стан і метрики:

```
# структура
get_campaigns(account_id="act_XXX", status_filter="ACTIVE")
get_adsets(account_id="act_XXX")
get_ads(adset_id="<adset_id>")              # перед діями над ad

# метрики за 5 періодів (як в ads-optimizer §3.3)
get_insights(object_id="act_XXX", time_range="today",     level="adset")
get_insights(object_id="act_XXX", time_range="yesterday", level="adset")
get_insights(object_id="act_XXX", time_range="last_3d",   level="adset")
get_insights(object_id="act_XXX", time_range="last_7d",   level="adset")
get_insights(object_id="<adset_id>", time_range="yesterday", level="ad")   # для ad-eater
```

Правила:
- **Числа не переносяться між сесіями як факт.** Те, що вчора CPL був $4, сьогодні треба перечитати. Пам'ять — це контекст, не джерело істини.
- **Read підтверджує і об'єкт, і його стан:** `object_id` ↔ `name` ↔ поточне значення (budget / status). Дублювати/паузити можна лише те, що реально прочитано.
- **`old_value` у плані зміни = прочитане значення**, не припущене.
- Якщо Marketing API повернув порожньо / помилку — це **не** «0 конверсій» і **не** «все ок». Це «дані не отримані» → стоп, повідомити, не діяти.

## 2. Dangerous-операції та формат підтвердження

### 2.1 Класифікація (зведено з `safety_rules.md`)

| Рівень | Операції | Що робити |
|---|---|---|
| **Критичні (завжди питати «так»)** | зміна бюджету (±), пауза campaign/adset/ad, створення кампанії/адсета, видалення будь-чого, зміна таргетингу | повний блок підтвердження → чекати «так» |
| **Середній ризик (показати план)** | resume paused-об'єкта, дублювання campaign/adset, зміна bid strategy | показати план + наслідок → чекати «так» |
| **Низький ризик (без підтвердження)** | get_insights, get_campaigns/adsets/ads, get_ad_creatives, get_custom_audiences, breakdowns, звіти, пошук інтересів/локацій | виконувати вільно (read-only) |

> **Включати (resume/activate) кампанії агент сам не вирішує остаточно — рішення вмикання за користувачем.** Кампанії з CBO і адсети з `lifetime_budget` не чіпати без явного запиту.

### 2.2 Обов'язковий формат підтвердження (для кожної критичної дії)

Перед mutate показати **таблицю змін** і чекати слова «так» (або «частково»/«ні»):

| Поле | Зміст | Приклад |
|---|---|---|
| `object` | тип + `object_id` + `name` | adset `adset_id=A` «Hook_video_v3» |
| `change` | **old → new** (точно) | budget $20 → $26 (+30%) |
| `обґрунтування (DATA)` | метрика + період + поріг | CPL $2.8 < target $4 (−30%), HS +35 (get_insights last_7d @2026-06-03) |
| `обґрунтування (KNOWLEDGE)` | принцип + `[[link]]` | масштабувати переможців помірно [[04-budget-scaling/big-spend-scaling]] |
| `очікуваний ефект` | з арифметикою, не «зросте» | +30% spend при тому ж CPL ≈ +6 reg/день |
| `ризик` | що може піти не так | re-entry в learning phase на ~24–48 год [[05-optimization/learning-phase]] |
| `відкат` | як повернути | update_adset назад на $20 |

**Без блоку підтвердження mutate-виклик не робиться.** «Очікуваний ефект» завжди прив'язаний до прочитаних цифр (еталон: §БЛОК ЗАХИСТУ у `fb-deep-audit-meta-prompt.md` — «погано: CTR зросте на 20%; добре: 5 креативів top-3 пар × +50% бюджет → CTR 2.5% → CPC $0.42 → $/Reg $2.40»).

### 2.3 API-виконання (тільки ПІСЛЯ «так»)

```
update_adset(adset_id="A", daily_budget=2600)   # бюджет у ЦЕНТАХ ($26)
pause_adset(adset_id="B")
pause_ad(ad_id="C")
# create_adset(...) — з урахуванням §3 (ліміти) і §4 (час)
```

## 3. Ліміти бюджету та межі

| Дія | Ліміт за раз | Причина |
|---|---|---|
| Збільшення бюджету | **+30% max** | різкий стрибок ламає навчання алгоритму (re-learning) |
| Зменшення бюджету | **−50% max** | надто різке зниження може «вбити» адсет |

| Параметр | Значення (якщо не задано брифом) |
|---|---|
| Мінімум бюджету адсета (**floor**) | **$3/день** (300 центів) |
| Максимум бюджету адсета | $100/день |
| Бюджет нового адсета | $10–20/день |

Правила:
- Розрахунок дав < $3 → ставити **$3** (floor), не нижче.
- Бюджети в центах; робочий діапазон **300…10000** ($3…$100).
- Приклад: поточний $20 → max+ = `20×1.30=$26`; max− = `20×0.50=$10`.
- При паузі/зниженні **звільнений бюджет перерозподіляється** (логіка `ads-optimizer` §9), а не «зникає».

> ⚠️ **Ліміти ±30%/−50% — це операційні запобіжники цього движка**, узгоджені з [[05-optimization/learning-phase]] (часті значні зміни рестартять learning). Це не дослівне правило Бена — Бен зміщує акцент із «% зміни» на **частоту будь-яких значних змін** (≤1 значна зміна / 7 днів). Тобто навіть зміна в межах +30% не робиться, якщо адсет правили <7 днів тому без критичної причини (див. §9).

## 4. Часові обмеження

- **Не створювати нові adsets після 18:00** за часовим поясом акаунта. Причина: алгоритм Meta не встигне оптимізувати за залишок дня. Виняток — явний терміновий запит користувача.
- **Learning phase (нові адсети ~24–48 год):** не міняти бюджет і не міняти таргетинг, поки адсет у learning, **окрім критичних проблем** (CPL > 3× target). Деталі й чому стабільність важлива → [[05-optimization/learning-phase]].
- Перед створенням адсета перевірити локальний час акаунта (з брифу / налаштувань), не системний час машини.

## 5. Пороги значущості (не діяти на шумі)

**Не судити об'єкт і не змінювати бюджет на недостатній вибірці.**

| Контекст | Метрика | Мінімум |
|---|---|---|
| Для зміни бюджету (yesterday/3d/7d) | Impressions | **1000** |
| | Конверсії | **3** |
| | Днів з даними | **2** |
| Для оцінки **сьогоднішнього** дня | Impressions | **300** |
| | Spend | **$3** |

Правила:
- Менше порога → **«недостатньо даних, чекаємо»**, а не рішення. Це і є чесна анти-галюцинаційна відповідь.
- Поріг значущості діє і у **звіті**: на адсеті/ad з <1000 показів або <3 конв. не писати «спрацював/провалився» — писати «вибірка мала, потрібно більше даних / `breakdowns=...`».
- Volume Factor у Health Score (×0.6…1.0) — це той самий принцип: малий обсяг знижує довіру до висновку, не скасовуючи його (див. `ads-optimizer` §5, компонент 5).
- Для перевірки «A справді кращий за B» (особливо малий бюджет) — statistical significance calculator, орієнтир ~90%+ [[04-budget-scaling/small-budget]].

## 6. Ad-Eater Detection (рівні дії за CPL)

**Ad-eater = конкретне ОГОЛОШЕННЯ (`ad`), що їсть бюджет без результату** (не адсет). Пороги відносно `target_CPL` з брифу:

| Рівень | Критерій | Дія |
|---|---|---|
| **CRITICAL** | CPL > **3× target** | негайна пауза `pause_ad` |
| **HIGH** | zero leads при spend ≥ 2× target | пауза |
| **HIGH** | CPL > **2× target** і spend_share ≥ 50% | пауза |
| **MEDIUM** | CPL > **1.5× target** при spend_share ≥ 50% | пауза з обережністю / моніторинг |

Логіка (як `ads-optimizer` §6):
1. `get_insights(object_id="<adset_id>", level="ad", time_range="yesterday")`.
2. `totalSpend` = сума spend усіх ad; знайти топ-спендера.
3. Якщо `topAd.spend ≥ 50% totalSpend` і його `CPL > target×1.3` → це ad-eater.
4. **Перевірити `will_adset_be_empty`:** якщо після паузи ad-eater(ів) адсет лишиться **порожнім** — не паузити ad окремо, а вирішувати долю адсета (pause_adset лише якщо HS ≤ −25).

Приклад (target $5): CRITICAL > $15; HIGH > $10 і spend_share ≥ 50%; MEDIUM > $7.50.

> Ad-eater — це **DATA-діагноз** (потрібні прочитані spend/leads/impressions). Коренева причина («чому їсть») — це вже **гіпотеза**: креатив / аудиторія / плейсмент; її перевіряти через breakdowns, а не стверджувати (див. §10).

## 7. Freshness guard (era → актуальність методології)

KNOWLEDGE еволюціонує. Перш ніж спиратись на тактику з `knowledge/`:

- Подивитись `era` у frontmatter файлу.
- Якщо `era: pre-andromeda` (або тактика про **таргетинг / аудиторії / тестування / структуру** з мітками «⚠️ перевір на актуальність») — **звірити з** [[00-philosophy/andromeda-era]] перед застосуванням. Можливо, тактика `superseded`.
- При конфлікті старе/нове перемагає **найсвіжіша тактика** (правило §5 META-PROMPT); стара лишається у блоці «Як було раніше».
- `status: superseded` → не застосовувати як актуальну пораду; згадати лише як «раніше було так».

**Типові пастки pre-Andromeda, що НЕ можна видавати за актуальне:**
- «≤6 оголошень/адсет» (зараз норма 20+ якісних різнопланових) [[00-philosophy/andromeda-era]].
- «жорстко 50 конверсій/7 днів інакше провал» (поріг не абсолютний; learning limited буває прибутковим) [[05-optimization/learning-phase]].
- «горизонтальний скейл = дублюй адсети під різні аудиторії» (пост-Andromeda частіше — консолідація) [[04-budget-scaling/small-budget]] · [[01-structure/campaign-structure]].

## 8. «Не знаю» дозволено і заохочується

Якщо даних у API-вибірці немає або в `knowledge/` немає принципу — **сказати прямо**, а не вигадати.

Шаблони чесної відповіді:
- «У наданій вибірці немає breakdown по плейсменту — щоб судити Stories vs Feed, потрібен `get_insights(..., breakdowns="publisher_platform")`. Зробити?»
- «Overlap аудиторій A↔B з цих даних не видно — це перевіряється через FB Audience Insights / окрему вибірку, не з get_insights.»
- «Бенчмарку CPM для цієї ніші в `knowledge/` немає — не вигадую число; орієнтуйся на власну медіану акаунта (get_insights last_30d).»

Заборонено:
- Вигадувати бенчмарки галузі («норм CPL для ніші $4»), якщо їх немає в DATA чи KNOWLEDGE.
- Стверджувати причину («Stories показали гірше») без breakdown, що це показує.
- Заповнювати прогалину «правдоподібним» числом. Прогалина → запит вибірки, а не вигадка.

Доступні breakdowns (через get_insights): `age`, `gender`, `country`, `device_platform`, `publisher_platform`.

## 9. History — читати до, логувати після

### 9.1 Читати перед оптимізацією (3 дні)

Перед будь-якими mutate-діями прочитати історію за **останні 3 дні**:

```
.claude/ads-agent/history/YYYY-MM/YYYY-MM-DD.md   # сьогодні, вчора, позавчора
```

З історії запам'ятати: які adsets/ads вже мінялись; коли створені нові об'єкти (для вікна learning 48 год).

### 9.2 П'ять правил історії (контекст для розумних рішень, не жорсткий блок)

| Ситуація | Правило | Виняток |
|---|---|---|
| Адсет створений < 48 год | не чіпати агресивно | CRITICAL ad-eater (CPL > 3× target) |
| Вчора вже знижували бюджет | не знижувати знову | CPL виріс у 3+ рази |
| 3 зниження за 3 дні | **пауза** замість чергового зниження | — |
| Вчора підвищували бюджет | не знижувати сьогодні (анти-коливання) | CPL виріс у 2+ рази |
| Today краще за yesterday | моніторинг замість зниження | — |

> Історія — **контекст**, не залізна заборона: при справді критичній ситуації (CPL 5× target) діяти негайно, попри історію. Але рішення «в обхід історії» теж обґрунтовується цифрами в блоці підтвердження.

### 9.3 Логувати після виконання

Після кожної серії дій дописати в `history/YYYY-MM/YYYY-MM-DD.md`:

```markdown
## HH:MM — Оптимізація {account} (skill: ads-optimizer)

| # | Тип | Object ID | Object Name | Old | New | Причина (DATA) | Статус |
|---|-----|-----------|-------------|-----|-----|----------------|--------|
| 1 | budget_increase | adset_id=A | Hook_v3 | $20 | $26 | CPL $2.8, HS +35 | success |
| 2 | pause_ad | ad_id=C | Bad_creative | active | paused | ad-eater, $25/0 leads | success |
```

Типи для логу: `budget_increase`, `budget_decrease`, `pause_ad`, `pause_adset`, `resume_ad`, `resume_adset`, `create_adset`, `create_campaign`, `update_targeting`.

### 9.4 При помилці виконання

1. Негайно повідомити користувача.
2. Спробувати **відкат** (повернути бюджет / resume paused).
3. Записати в history, що сталося.

## 10. DATA → діагноз → гіпотеза (як не видати здогад за факт)

Ланцюг висновку має три чітко розмежовані ланки:

1. **DATA (факт):** «adset_id=A CTR впав 2.7% → 2.2% за 14 днів (get_insights, daily, @2026-06-03)». Перевірено, з джерелом.
2. **Діагноз (правдоподібно з DATA + KNOWLEDGE):** «frequency_7d=3.1 + падіння CTR → ознака creative fatigue [[06-creatives/why-ads-fail]]».
3. **Гіпотеза (потребує перевірки):** «ймовірно, виноситься на Stories — перевірити `breakdowns=publisher_platform`».

Правило: у звіт/рекомендацію ланка 3 йде **з поміткою «гіпотеза, перевірити через …»**, а не як факт. Кожна рекомендація = конкретний `object_id` + метрика + поріг + дія (не «покращити креативи», а «pause ad_id=C: CPL $18 = 3.6× target $5; дублювати ad_id=Y, +30% бюджет»).

## ЧЕК-ЛИСТ перед dangerous-операцією

Виконати **всі** пункти; якщо хоч один «ні» — стоп.

- [ ] **Read-before-write:** поточний стан і метрики об'єкта прочитані з Marketing API **цієї сесії** (§1).
- [ ] **Правильний акаунт:** `act_XXX` збігається з брифом (§2).
- [ ] **Правильний об'єкт:** `object_id` ↔ `name` звірені (§1).
- [ ] **Бриф прочитано:** `target_CPL` / `target_ROAS` / бюджети взяті з брифу, не вигадані (§3).
- [ ] **Історія за 3 дні прочитана**, 5 правил застосовано (§9.1–9.2).
- [ ] **Достатньо даних:** ≥1000 показів / ≥3 конв. / ≥2 дні (або 300/$3 для today) (§5).
- [ ] **Ліміти дотримано:** ≤+30% / ≥−50%; не нижче $3 floor; не вище $100 (§3).
- [ ] **Час:** не створюю новий adset після 18:00 без явного запиту (§4); адсет не в learning <48 год (або причина критична) (§4, §9).
- [ ] **Freshness:** тактики з `knowledge/` не `pre-andromeda`/`superseded` без звірки з [[00-philosophy/andromeda-era]] (§7).
- [ ] **KNOWLEDGE ≠ DATA:** принцип і спостереження розмежовані; гіпотези помічені «перевірити через …» (§0, §10).
- [ ] **Блок підтвердження показано:** object_id, old→new, обґрунтування DATA+KNOWLEDGE, очікуваний ефект (з арифметикою), ризик, відкат (§2.2).
- [ ] **Отримано «так»** від користувача.
- [ ] _(після виконання)_ **Дії залоговані** в `history/` (§9.3).

## SELF-CHECK (як перевірити / фальсифікувати дотримання цих правил)

Перед фінальним звітом або після серії дій пройти ці тести; будь-який провал = повернутись і виправити.

1. **Тест джерела кожної цифри.** Візьми 5 випадкових чисел про кабінет зі звіту. Для кожного назви `object_id` + період + API-виклик, з якого воно взяте. Якщо хоч одне не трасується до read цієї сесії — це галюцинація, прибрати (§1).
2. **Тест KNOWLEDGE/DATA.** Пройди по кожному «Бен каже / принцип…». Чи не видано його за спостереження конкретного акаунта? Кожне «у тебе виснаження/виграш» має поряд DATA-доказ із датою (§0).
3. **Тест порога значущості.** Чи є висновок «спрацював/провалився» на об'єкті з <1000 показів або <3 конв.? Якщо так — замінити на «мала вибірка, потрібно більше даних / breakdowns» (§5).
4. **Тест дієвості.** Кожна рекомендація має `object_id` + метрику + поріг + конкретну дію? «Покращити креативи» без id — провал; переписати як «pause ad_id=X (CPL 3× target)» (§10).
5. **Тест лімітів.** Чи всі запропоновані зміни бюджету в межах +30%/−50% і ≥$3? Чи немає створення adset після 18:00 без явного запиту? (§3, §4)
6. **Тест freshness.** Чи всі застосовані тактики мають `era: post-andromeda` / `evergreen`, або звірені з [[00-philosophy/andromeda-era]]? Pre-Andromeda без звірки — провал (§7).
7. **Тест «не знаю».** Чи є у звіті число/причина, яких немає ні в DATA, ні в KNOWLEDGE (вигаданий бенчмарк, причина без breakdown)? Якщо так — замінити на запит вибірки (§8).
8. **Тест відкату.** Для кожної критичної зміни — чи описано, як її відкотити? Якщо відкат неможливий/незрозумілий — підняти ризик і ще раз спитати користувача (§2.2, §9.4).
9. **Тест ≥5 слабких місць.** Сформулюй мінімум 5 тверджень «цей висновок хибний, якщо …» (напр.: «якщо атрибуційне вікно інше, ніж я припустив», «якщо conversions ще не догналися через затримку Pixel», «якщо breakdown по плейсменту перевернув би картину»). Хороший аналіз знає, де його гіпотеза слабка (еталон §self-check у `fb-deep-audit-meta-prompt.md`).
10. **Тест історії.** Чи прочитана історія за 3 дні і чи не порушено жодне з 5 правил без зафіксованої критичної причини? Чи заплановано лог виконаних дій? (§9)

> Якщо документ/звіт проходить усі 10 тестів — рішення безпечні й трасовані. Якщо ні — це не «майже готово», а «повернутись до §-винуватця».
